## 面向人类的使用指引（需求分析部分）

### 你需要做什么
1. 先用自然语言说出 Feature 想法（不用一次说完整）。
2. 按提问逐步补齐：目标用户、核心场景、边界、异常、非功能要求。
3. 对每轮总结给出「确认 / 修正 / 补充」三选一反馈。
4. 在进入下一阶段前，明确给出放行结论（Go / No-Go）。

### 三阶段执行顺序（认知版）
1. **收敛边界**：把“想法”变成可验证的业务规则与约束。
2. **架构拆解**：把规则拆成可执行 Task、依赖和优先级。
3. **契约固化**：把验收标准转成测试契约（如 Gherkin / TDD 输入）。

### 每阶段你应重点检查什么
- **阶段一产物（业务规则备忘录）**：术语是否统一、边界是否清晰、异常是否覆盖、指标是否可量化。
- **阶段二产物（Task 执行列表）**：任务粒度是否可交付、依赖是否完整、验收标准是否可测。
- **阶段三产物（测试契约）**：场景是否覆盖主流程/边界流/失败流，是否可直接驱动开发与验收。

### 常见误区（请避免）
- 只给“功能清单”，不给边界与异常处理。
- 用“高性能/高可用”这类空泛描述，不给可测阈值。
- 跳过阶段门禁，直接进入开发实现。
- 需求变更后不回刷前序产物，导致契约与实现脱节。

### 与本仓库 Skill 的衔接（统一执行链）
- 需求分析完成后，必须按顺序进入本地四步链：
  - `forge_requirements`：落盘需求契约产物。
  - `enforce_tdd`：执行 TDD 门禁并沉淀证据。
  - `trigger_blind_review`：独立盲审。
  - `submit_to_owner`：最终放行归档。
- 禁止跳步：任一环节失败都必须回退修复后重试。

### 按时机触发的 Skill 调用清单（谁在何时调用什么）

| 所属阶段 | 时机 | 人类要做什么 | 应调用的 Skill | 调用方 |
|---|---|---|---|---|
| 需求分析-阶段1 收敛边界 | 用户刚提出 Feature 想法（模糊） | 提供业务目标与背景 | `reverse-interviewing` | Agent 主动调用（人类确认回答） |
| 需求分析-阶段1 收敛边界 | 第一轮问题后仍有歧义/冲突 | 补充场景、边界、角色诉求 | `interview-conducting` | Agent 主动调用（人类继续答复） |
| 需求分析-阶段1 收敛边界 | 已有多来源输入，需要整合和缺口分析 | 确认冲突裁决与优先级 | `elicitation-methodology` | Agent 主动调用（人类做 Go/No-Go） |
| 需求分析-阶段2 架构拆解 | 业务规则备忘录已确认，准备技术拆解 | 确认技术约束与目标架构方向 | `architecture-patterns` | Agent 主动调用（人类确认） |
| 需求分析-阶段2 架构拆解 | 架构方案已确认，准备排执行计划 | 确认里程碑、依赖、优先级 | `decomposition-planning-roadmap` | Agent 主动调用（人类确认） |
| 需求分析-阶段3 契约固化 | Task 列表定稿，准备测试契约 | 确认关键验收场景与失败流 | `tdd` | Agent 主动调用（人类确认） |
| 需求分析后衔接（本地链路起点） | 进入本地仓库落盘阶段 | 确认 task_id 与需求文本版本 | `forge_requirements` | 人类触发 CLI |
| 开发执行阶段 | 进入编码门禁阶段 | 指定测试命令并执行门禁 | `enforce_tdd` | 人类触发 CLI |
| 审查阶段 | 代码通过本地门禁后 | 发起独立盲审 | `trigger_blind_review` | 人类触发 CLI |
| 发布裁决阶段 | 盲审通过后 | 做最终发布裁决 | `submit_to_owner` | 人类触发 CLI（交互/非交互） |

> 推荐原则：阶段一到三以 Agent 编排为主，人类负责确认；从 `forge_requirements` 开始进入本仓库的本地执行链，由人类显式触发 CLI。

### 保留/抛弃决议（技能层）
- 保留：`forge_requirements` / `enforce_tdd` / `trigger_blind_review` / `submit_to_owner`
- 抛弃：`read_task_contract` 与任何“手工跨窗口日志搬运”做法
- 原因：前者可形成端到端门禁闭环，后者会造成证据错位和流程漂移
