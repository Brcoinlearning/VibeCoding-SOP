# 单一主仓库重组方案

## 文档定位

本文档只回答一件事：**当前项目如何从“外层仓库 + 内层 superpowers 仓库”的过渡状态，演化为一个新的、单一主仓库管理的 superpowers 风格产品。**

它定义的是仓库治理原则、吸收策略与迁移阶段，不直接在本轮执行完整重组，也不在当前阶段提前删除尚未确认无用的上游内容。

## 当前理解

当前项目与原 `superpowers` 的关系，不应继续理解为：

- 在外层做一个增强层项目
- 在里面内嵌一个 `superpowers`

更准确的理解应是：

- 当前仓库正在演化为一个**新的 superpowers 风格产品**
- 原 `superpowers` 是当前最重要的上游参考来源与吸收对象
- `superpowers/` 目录当前仍保留，是为了完整研究、吸收、迁移其内容，而不是作为长期最终结构的一部分

这意味着：

- 最终不应形成“软件开发SOP 外层 + superpowers 内层”这种双层产品表达
- 最终产品本体应直接继承 `superpowers` 的仓库组织方式与技能系统表达
- 当前阶段应优先解决的是“如何完整吸收”，而不是“如何尽快删掉子仓库目录”

## 当前结构存在的问题

当前仓库的主要问题不是内容不够，而是边界表达错误：

1. 外层仓库已经承载了本项目的正式需求、架构、契约与新增 skills。
2. 内层 `superpowers/` 仍然保存了大量上游核心能力，而且部分关键 skill 已被本项目改写。
3. 当前真正参与正式流程的能力，分散在外层 `skills/` 与内层 `superpowers/skills/` 两处。
4. `brainstorming` 与 `writing-plans` 已不再是当前项目默认主链，但原仓库中仍保留它们的完整结构与参考价值。
5. 当前如果过早清理 `superpowers/`，容易误删尚未识别价值的上游内容。

因此，当前阶段不适合直接做“删空内层仓库”式的收敛，而应先完成完整吸收与映射。

## 重组结论

### 核心结论

本项目后续按以下方向推进：

**以外层仓库为最终唯一主仓库，逐步吸收原 `superpowers` 的完整有价值内容，并最终演化为一个新的 superpowers 风格仓库。**

这句话包含四层含义：

1. 最终 git 管理边界只有一个，即外层仓库。
2. 最终产品表达应贴近 `superpowers` 的根目录风格，而不是长期保留“外层包内层”的结构。
3. 当前阶段仍保留 `superpowers/` 目录，作为完整参考资产容器。
4. 只有在完整吸收、确认保留范围之后，才进入最终退场阶段。

### 为什么不是增强层项目

本项目已经不是“在原产品外面补几层规则”，而是在重写默认主流程：

- 前半主链已改为四阶段准备链
- `Architecture` 已直接承担 task 来源职责
- 正式开发透明度已成为项目级门禁
- reviewer 隔离已成为项目级门禁
- `brainstorming` / `writing-plans` 已不再是默认主链步骤

这类变化已经超出轻量 overlay 的合理边界，更符合“新产品演化版”的事实。

## 当前阶段的首要原则

### 原则 1：先完整吸收，再决定删除

在没有确认 `superpowers/` 中哪些内容真正无用之前：

- 不着急删除目录
- 不着急清空次级文件
- 不把“当前主流程暂时没用到”误判成“最终没有价值”

### 原则 2：先保留原仓库整体形态，再逐步吸收

当前对 `superpowers` 的利用，不应只盯住：

- `skills/`
- `commands/`
- `agents/`
- `docs/`
- `tests/`

虽然这些目录是显性的核心层，但原仓库中仍可能有其他结构、辅助文件、配置文件、说明文件在后续迁移时具有价值。

因此当前策略是：

- 尽量原封不动保留原仓库内容
- 先做完整盘点与吸收映射
- 再决定最终保留范围

### 原则 3：当前最先要终结的是 nested git，不是目录本身

从治理角度看，当前最严重的问题是：

- 有两个 git 边界
- 有两个提交与版本管理语义

所以优先级应该是：

1. 先解决“内层仓库不应继续作为独立 git 仓库”这个问题
2. 再解决“目录最终是否保留、如何归并”这个问题

也就是说：

- `superpowers/.git` 最终必须退场
- 但 `superpowers/` 目录在迁移期可以继续存在

## 最终产品的结构方向

当前不提前写死最终完整目录树，但明确以下结构方向：

### 方向 1：最终仓库应继承 superpowers 的根目录表达方式

也就是说，最终产品应尽量表现为一个新的 superpowers 风格仓库，而不是继续保留“外层项目 + 内层 superpowers”的表达。

这意味着：

- `skills/` 仍然会是核心目录
- `commands/`、`agents/`、`docs/`、`tests/` 等目录层次也应优先参考原 `superpowers` 的产品表达方式
- 当前外层临时形成的调研期目录，不应被直接当作最终产品结构模板

### 方向 2：当前 `10/15/20/25` 文档层最终应服从产品级文档组织方式

当前阶段文档已经按 `10/15/20/25` 稳定下来，这是好的。

但它们本质上是：

- 需求文档
- 技术选型文档
- 架构文档
- 契约文档

因此后续它们是否继续保留在根目录，还是被重新纳入产品级 `docs/` 结构，应在完整重组阶段再决策。

当前阶段不提前下结论。

### 方向 3：当前新增能力最终要内生化为新产品的一部分

以下能力不应长期作为“外挂层”存在，而应最终成为新产品的正式组成部分：

- `boundary-convergence`
- `tech-selection`
- `architecture-and-tasking`
- `contract-solidification`
- `development-transparency-protocol`
- 已被改写的正式开发相关 skills

也就是说，它们最终不是“套在 superpowers 外面的东西”，而是“新版本 superpowers 的内生能力”。

## 迁移期结构理解

在真正完成吸收之前，当前允许存在如下过渡性结构：

```text
软件开发SOP/
├── 10-requirements/
├── 15-tech-selection/
├── 20-architecture/
├── 25-contract/
├── skills/
│   ├── boundary-convergence/
│   ├── tech-selection/
│   ├── architecture-and-tasking/
│   ├── contract-solidification/
│   ├── development-transparency-protocol/
│   └── ...
├── superpowers/
│   ├── skills/
│   ├── commands/
│   ├── agents/
│   ├── docs/
│   ├── tests/
│   └── ...
└── .git
```

这个结构的含义不是“最终就这样”，而是：

- 外层仓库已经承载新的定制方向
- 内层目录仍保留完整上游内容，供系统性吸收
- 当前阶段先做吸收映射，不先做大规模裁剪

## Skill 吸收策略

### A 类：当前主链必需能力

以下能力已经被确认属于当前主链，后续必须被正式吸收到新产品结构中：

- `boundary-convergence`
- `tech-selection`
- `architecture-and-tasking`
- `contract-solidification`
- `development-transparency-protocol`
- `using-git-worktrees`
- `subagent-driven-development`
- `test-driven-development`
- `requesting-code-review`
- `executing-plans`
- `finishing-a-development-branch`

### B 类：非主链但暂不删除的参考能力

以下能力当前已不再属于默认主链，但仍保留明显参考价值：

- `brainstorming`
- `writing-plans`

当前处理策略：

1. 不作为默认主链步骤继续宣传
2. 不在当前阶段草率删除
3. 后续可标记为非主链参考能力或 legacy 能力

### C 类：待盘点能力

除上述明确主链能力外，原 `superpowers` 其他内容当前都应先进入“待盘点”状态，而不是直接丢弃。

盘点维度包括：

- 是否仍有直接运行价值
- 是否有结构参考价值
- 是否有提示词/模板参考价值
- 是否有测试/验证参考价值

## `superpowers/` 目录的阶段性角色

### 当前角色

当前 `superpowers/` 的角色是：

- 完整上游参考仓库内容容器
- 当前尚未完全吸收的核心资产来源
- 迁移期间的事实依据

### 迁移期要求

迁移期间：

- 保留目录整体内容
- 不急于删减目录层次
- 所有吸收动作都应形成映射记录

### 最终角色

当以下条件全部满足时，`superpowers/` 才进入退场决策：

1. 已完成完整吸收盘点
2. 已确认主链能力全部迁移或内生化
3. 已确认非主链但保留内容的归宿
4. 已不再依赖 nested git 边界工作

在此之前，不提前进入“删整个子仓库目录”的动作。

## Git 治理原则

### 原则 1：最终只有一个正式 git 仓库

后续正式开发、提交、发布，都应基于外层仓库进行。

### 原则 2：`superpowers/.git` 必须退场

无论目录是否暂时保留，内层独立 git 边界都不应长期存在。

### 原则 3：来源关系通过文档记录

与上游 `superpowers` 的关系，应通过文档表达，例如：

- README 来源说明
- upstream diff 文档
- skill 差异映射文档
- 迁移决策文档

## 迁移阶段设计

### 阶段 1：方向确认阶段

目标：确认“不是增强层，而是新的 superpowers 风格产品”。

本阶段完成标准：

- 明确单一主仓库方向
- 明确 `superpowers/` 当前保留、最终退场的原则
- 明确不采用“先删后补”的激进迁移方式

### 阶段 2：完整吸收盘点阶段

目标：完整识别 `superpowers/` 中哪些内容有用，以及它们未来的去向。

本阶段重点：

- 全量盘点 `superpowers/` 目录内容
- 不只盘点 `skills/`，也盘点 `commands/`、`agents/`、`docs/`、`tests/` 等内容
- 建立“保留 / 改写 / 替换 / 暂不使用”的映射清单

完成标准：

- 已形成完整吸收地图，而不是靠印象判断哪些内容有用

### 阶段 3：主链能力收拢阶段

目标：把已经确认保留的主链能力逐步收拢进新的产品主结构。

本阶段重点：

- 迁移主链必需 skills
- 迁移依赖的 prompt 与 reference 文件
- 同步修正 README、架构文档、skill map、契约文档引用

完成标准：

- 当前主流程运行不再依赖 nested git 边界

### 阶段 4：非主链能力定位阶段

目标：处理不再属于默认主链、但仍可能保留参考价值的上游能力。

本阶段重点：

- `brainstorming` 的定位
- `writing-plans` 的定位
- 其他非主链能力的保留方式

完成标准：

- 已区分主链能力、参考能力、可移除能力

### 阶段 5：子仓库退场阶段

目标：在完成吸收之后，终结 `superpowers/` 的内层仓库语义。

本阶段重点：

- 清理 nested git 关系
- 根据吸收结果决定目录归并方式
- 补齐上游差异记录

完成标准：

- 外层仓库成为唯一正式产品仓库
- 目录语义与 git 边界一致

## 风险与控制策略

### 风险 1：过早清理导致丢失尚未识别价值的上游内容

控制策略：

- 先全量盘点
- 先做映射，再做删除决策

### 风险 2：只看表层目录，忽视原仓库其他有价值内容

控制策略：

- 盘点范围覆盖整个 `superpowers/` 仓库
- 不只关注 `skills/`、`commands/`、`agents/`、`docs/`、`tests/`

### 风险 3：迁移过程中 README 与架构文档继续失配

控制策略：

- 每一轮结构迁移后，都同步检查 README、架构文档、skill map、execution-progress、test contract

### 风险 4：过早删除 `brainstorming` / `writing-plans` 导致失去参考价值

控制策略：

- 当前先降级，不草率删除
- 等主链稳定后再决定最终归宿

## 与现有文档的关系

本文档补充的是**仓库治理与重组方案**，不是替代以下文档：

- `README.md`：项目总览
- `20-architecture/architecture.md`：目标流程架构
- `20-architecture/superpowers-deep-customization-skill-map.md`：skill 变更映射
- `20-architecture/execution-progress.md`：当前执行进度

它回答的是：

- 当前仓库与原 `superpowers` 的关系应该如何理解
- 为什么当前阶段要先完整吸收而不是先大删目录
- 后续重组任务应该分几步实施

## 当前阶段结论

当前已经达成的共识是：

1. 本项目后续按**单一主仓库**方向推进。
2. 本项目的目标不是“内嵌一个 superpowers”，而是演化为一个新的 superpowers 风格产品。
3. 当前阶段先完整保留并吸收 `superpowers/` 的有价值内容，不着急删除目录本身。
4. 当前最先需要终结的是 nested git 状态，而不是先把所有目录删平。
5. `brainstorming` 与 `writing-plans` 不再是默认主链步骤，但在当前阶段仍保留参考价值，不草率删除。
6. 完整重组任务应在后续单独实施，不与当前文档收敛工作混做。
