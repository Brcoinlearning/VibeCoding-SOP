# 03 对抗审查与门禁

## 3.1 设计目标

缓解“礼貌偏置”造成的漏检问题，但不制造无意义挑刺。

## 3.2 审查提示词策略

- 使用对抗角色设定，禁止泛化夸赞。
- 强制输出结构化 JSON。
- 强制给出可复现路径与影响面。

示例输出格式：

```json
{
  "status": "REJECTED",
  "severity": "BLOCKER",
  "lethal_flaw": "并发写入导致库存负数",
  "exploit_path": "两个并发请求同时扣减同一库存",
  "evidence": ["test_case: test_concurrent_deduct", "log_ref: run-20260410-1130"],
  "fix_hint": "增加原子更新或事务隔离"
}
```

## 3.3 最低发现义务

Reviewer 每轮至少完成以下检查：
1. 并发/时序风险检查。
2. 错误处理与回滚检查。
3. 时区/时间边界检查。
4. 验收条件覆盖检查。

如果四项均无问题，必须给出“检查证据”，而不是口头“看起来没问题”。

## 3.4 门禁

- Blocker 未关闭：No-Go。
- Major 未处理：需要 Owner 显式风险接受。
- 未执行完成前验证：禁止声明可合并。

## 3.5 反形式主义约束

为避免 Reviewer 为了满足“至少找一个问题”而胡乱挑刺，新增约束：
- 每个问题必须可复现。
- 每个问题必须绑定证据。
- 不可复现问题默认降级为观察项，不阻断。
