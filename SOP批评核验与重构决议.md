# SOP 批评核验与重构决议（2026-04-10）

## 1. 结论先行

本次外部批评中，存在三类内容：
- A 类（属实且高价值）：需要直接纳入 SOP 并强制执行。
- B 类（方向正确但表述绝对化）：保留技术方向，修正为可操作工程规则。
- C 类（结论过度或“唯一有效”叙事）：不采纳绝对化说法，改用可验证机制组合。

本决议用于约束后续执行，避免“情绪化反驳”或“全盘照抄”。

## 2. 逐条核验

### 2.1 关于“人工搬运导致伪 Agent”

判定：A 类（基本属实）。

核验：
- 手工跨窗口复制日志和 diff，确实会引入延迟、遗漏和操作疲劳。
- 但“完全无人值守”并非总是最优，关键决策仍需 Owner 负责。

处置：
- 采纳事件驱动交接。
- 保留人工在两处介入：异常中断、最终放行。

### 2.2 关于“仅隔离无法解决模型互相放水”

判定：B 类（方向正确，表述过度）。

核验：
- 仅做会话隔离，不能保证审查严格度。
- 但“强制负面 JSON 是唯一有效手段”不成立。还需结合静态检查、测试门禁、证据新鲜度校验。

处置：
- 采纳对抗性审查提示词与结构化输出。
- 增加“最少发现义务 + 失败回退 + 证据绑定”机制，避免形式化挑刺。

### 2.3 关于“长日志导致注意力退化”

判定：A 类（属实）。

核验：
- 原始大段日志直接输入会稀释重点，增加误判风险。
- 需要先提炼失败切片，再喂给 Reviewer。

处置：
- 采纳上下文裁剪、错误摘要、diff 限幅、标签分段。
- 规定“优先结构化摘要，原文按需追溯”。

### 2.4 关于“LLM 不适合精确路径管理”

判定：B 类（方向正确，绝对化不成立）。

核验：
- 路径拼写与映射维护确有概率失误。
- 但通过脚本托管、清单校验和固定命名规则，可将风险降到可控。

处置：
- 采纳脚本托管 I/O 与 frontmatter 路由。
- 不再要求 Agent 维护深层目录映射文档。

## 3. 重构原则（强制）

1. 人机边界重划：Agent 负责推理与生成；脚本负责交接与落盘。
2. 审查机制升级：从“礼貌建议”改为“对抗审查 + 证据约束 + 失败可追溯”。
3. 上下文工程化：禁止原始大日志直喂；必须摘要后分段输入。
4. I/O 决定性：路径、命名、归档由脚本决定，Agent 只给内容与元数据。
5. 放行责任单点化：仅 Owner 有 Go/No-Go 权限。

## 4. 对外表述口径

推荐使用以下口径：

“我们采纳了批评中关于自动交接、对抗审查、上下文裁剪和 I/O 托管的有效部分，并把绝对化表述改造成可验证、可审计、可执行的工程机制。重构后的 SOP 不是口号，而是可落地的运行协议与门禁系统。”
